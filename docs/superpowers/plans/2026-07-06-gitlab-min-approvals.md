# GitLab minimum-approvals threshold — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let each GitLab project require a minimum number of MR approvals (default 3) before AO treats the MR as `approved`, applied only when the GitLab repo has no approval rule of its own.

**Architecture:** Adapters stay dumb fact-reporters. The GitLab adapter reports two new raw facts per MR — the approval count and whether GitLab enforces an approval rule — persisted on the `pr` row. The min-approvals *policy* is applied at status-derivation time in the session service, where the per-project config is available. No GitHub behaviour changes.

**Tech Stack:** Go backend (SQLite via sqlc), GitLab REST v4, React/TypeScript renderer with an OpenAPI-generated client.

## Global Constraints

- App state lives under `~/.ao` only — do not touch other data locations.
- SQLite access goes through sqlc: after editing `*.sql`, regenerate with `npm run sqlc` from the repo root; never hand-edit `internal/storage/sqlite/gen/*`.
- After changing `domain.ProjectConfig` or any API type, regenerate the OpenAPI spec with `npm run api:spec` from the repo root so the renderer's `components["schemas"]` stays in sync.
- Threshold semantics: `approvals_count >= minApprovals`. `minApprovals <= 0` resolves to the default `3`.
- Fallback only: the threshold applies only when `ApprovalRuleConfigured == false`. A GitLab MR whose repo has an approval rule keeps GitLab's own decision, unchanged.
- GitLab only in this version; GitHub leaves the new facts zero-valued, which the threshold branch treats as a no-op.
- Run backend tests with `cd backend && go test ./...`; frontend tests with `npm test` from `frontend/`.

```go
// approvalDecision maps GitLab's approvals payload onto AO's normalized review
// decision. When GitLab enforces an approval rule, "approved" means no approvals
// remain outstanding and at least one is recorded. When there is NO rule
// (approvals_left is trivially 0), GitLab cannot express a floor, so we return no
// decision and let AO's per-project MinApprovals threshold decide from
// ApprovalsCount at status-derivation time.
func approvalDecision(a restApprovals) string {
	if !approvalRuleConfigured(a) {
		return ""
	}
	if a.ApprovalsLeft == 0 && len(a.ApprovedBy) > 0 {
		return "approved"
	}
	return ""
}

// approvalRuleConfigured reports whether GitLab enforces an approval rule of its
// own for this MR.
func approvalRuleConfigured(a restApprovals) bool {
	return a.ApprovalsRequired > 0 || a.HasApprovalRules
}
```

```go
func prPipelineStatus(pr domain.PRFacts, minApprovals int) domain.SessionStatus {
	switch {
	case pr.CI == domain.CIFailing:
		return domain.StatusCIFailed
	case pr.Draft:
		return domain.StatusDraft
	case pr.Review == domain.ReviewChangesRequest || pr.ReviewComments:
		return domain.StatusChangesRequested
	case pr.Mergeability == domain.MergeMergeable:
		return domain.StatusMergeable
	case pr.Review == domain.ReviewApproved:
		return domain.StatusApproved
	case pr.Review == domain.ReviewRequired:
		return domain.StatusReviewPending
	case !pr.ApprovalRuleConfigured && pr.ApprovalsCount >= minApprovals:
		// No SCM rule of its own: AO's per-project floor decides readiness.
		return domain.StatusApproved
	default:
		return domain.StatusPROpen
	}
}
```
