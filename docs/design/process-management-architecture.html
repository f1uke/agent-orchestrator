Issue #1848

Process lifecycle

AO Process Management Architecture

This document explains the process tree that ao start creates, the consumers of the managed-child interface, and why the interface is designed to prevent zombie or lingering processes during ao stop, SIGTERM, and Ctrl+C.

Status: PR design documentation
Date: 2026-05-15
Primary code: packages/core/src/daemon-children.ts
Consumers: CLI daemon and dashboard server launcher

Problem statement

AO has a long-running foreground daemon started by ao start. That daemon owns long-lived subprocesses: dashboard servers, websocket servers, lifecycle supervisors, and watcher processes. Node does not guarantee that a signal sent to the parent is propagated to its children. If the parent exits without explicitly terminating children, the kernel reparents them to init (PPID=1), where they keep file descriptors, TCP connections, and sometimes ports alive.

The architectural goal is not merely “remember to kill this one child.” The goal is that daemon-owned subprocesses have a formal creation interface, and using that interface makes leaking children difficult by default.

Process taxonomy

Process kind Owner Lifetime Cleanup mechanism
ao start daemon User shell or service manager Runs until Ctrl+C, ao stop, or external SIGTERM. installShutdownHandlers() performs full daemon cleanup and exits. ao stop also sends SIGTERM/SIGKILL to the daemon process tree.
Dashboard launcher child ao start Long-running. Starts either web dev mode or the production start-all.js. Spawned with spawnManagedDaemonChild("dashboard", ...), recorded in the daemon-child registry, and killed on stop/shutdown.
Next.js server Dashboard launcher Long-running web UI process. The dashboard launcher is also a managed-process consumer and spawns Next through spawnManagedDaemonChild() so a direct launcher SIGTERM or registry sweep reaches it.
Terminal websocket / mux websocket servers Dashboard launcher Long-running terminal transport. Managed child spawn from packages/web/server/start-all.ts. Historical unregistered variants are handled by the startup orphan sweep pattern matcher.
Lifecycle workers / project supervisors ao start daemon services Long-running project-level background supervision. Stopped in the daemon shutdown path through existing supervisor APIs; matching historical orphan commands are reaped during ao start --reap-orphans.
.ao-messaging/*/ao-msg-watch.mjs watchers Messaging/session helpers Long-running per-project or per-session watcher. Covered by AO orphan detection and should use the managed-child interface when spawned by daemon-owned code.
Agent runtime/session processes Runtime plugins and session manager Session-scoped. Existing session-manager/runtime plugin cleanup. This PR does not replace runtime ownership semantics; it handles daemon-owned infrastructure subprocesses.

Dependency chain

The simplified POSIX process tree below is the shape that caused the leak when the parent exited without signal forwarding. Windows uses taskkill /T /F behind the platform abstraction rather than POSIX process groups, but the ownership model is the same.

User shell

Runs ao start or sends Ctrl+C.

ao start

Writes running.json, installs shutdown handlers, starts daemon services.

Managed children

Dashboard launcher, project supervisors, lifecycle and watcher subprocesses.

Grandchildren

Next.js, websocket servers, terminal transports, and helper processes.

user shell
└─ ao start daemon
   ├─ dashboard launcher (managed)
   │  ├─ next-server (managed by launcher)
   │  └─ direct-terminal-ws / mux ws (managed by launcher)
   ├─ lifecycle-worker <project> (daemon service)
   └─ .ao-messaging/.../ao-msg-watch.mjs (watcher/helper)

Formal interface

The interface is intentionally small. It sits in core because both CLI and web server code need the same lifecycle semantics, and because future daemon-owned subprocesses should not invent their own cleanup model.

spawnManagedDaemonChild(role, command, args, options)

Consumer: code that creates a long-running daemon-owned child.

This is the preferred spawn primitive. It delegates to child_process.spawn, registers the child PID, installs signal forwarding, unregisters on child exit, and records a human-readable role/command for diagnostics.

registerChildReaper(child, role, command?)

Consumer: code that receives a ChildProcess from another API.

This is the lower-level adapter. It exists so callers do not need to bypass the registry if a child was spawned by a wrapper or library.

markDaemonShutdownHandlerInstalled()

Consumer: processes with their own async shutdown sequence.

The managed reaper still forwards signals to children, but it does not force process exit. The owning shutdown handler exits only after its async cleanup completes.

sweepDaemonChildren({ ownerPid })

Consumer: ao stop, daemon shutdown, and daemon replacement.

Reads the persistent registry, filters to the daemon owner PID when supplied, sends SIGTERM, waits within a bounded grace window, SIGKILLs stragglers, and prunes dead entries.

Stop and Ctrl+C semantics

ao stop

  1. Read running.json to find the daemon.
  2. Read daemon-children.json and sweep child PIDs owned by the running daemon PID.
  3. Signal the daemon process tree through killProcessTree().
  4. Escalate to SIGKILL if the daemon does not exit.
  5. Remove stale registry entries after confirmed process death.

Ctrl+C / SIGTERM on ao start

  1. installShutdownHandlers() marks ownership of async shutdown.
  2. Supervisor and lifecycle services are stopped first.
  3. Sessions are killed and last-stop metadata is written.
  4. Registered daemon children are swept.
  5. State files are unregistered and the process exits.

If a process uses managed children but does not install an app-level shutdown owner, the core fallback still performs SIGTERM → bounded wait → SIGKILL before exiting. That keeps the safe interface safe even for simpler consumers.

Persistent registry

The registry is stored at ~/.agent-orchestrator/daemon-children.json. Entries are deliberately minimal: 

{
  "pid": 90350,
  "role": "dashboard:next",
  "parentPid": 90236,
  "startedAt": "2026-05-15T00:00:00.000Z",
  "command": "node dist-server/start-all.js"
}

This file is not the source of truth for a child being alive; process liveness is always re-checked before returning or pruning entries. Its purpose is recovery after a parent crash, where in-memory signal handlers never got a chance to run.

Orphan recovery on start

ao start also scans for historical AO children whose parent is already PPID=1. This handles machines that accumulated leaks before the registry existed, or after an uncatchable parent death such as SIGKILL.

  • node .../@aoagents/ao-web@*/dist-server/...
  • node .../ao lifecycle-worker ...
  • node .../ao-messaging/.../ao-msg-watch.mjs
  • node ...next-server...

In an interactive shell, AO prompts before killing these processes. In automation, --reap-orphans opts in explicitly.

Design constraints and boundaries

Centralized platform behavior

Callers use killProcessTree() and isWindows() from core. They do not inline process.platform or shell out independently.

Lint-enforced safe path

Daemon entrypoints cannot import raw spawn from child_process. They must use the managed-child API for daemon-owned subprocesses.

Not a session runtime replacement

Runtime plugins still own session processes. This architecture covers infrastructure children owned by the daemon and dashboard launcher.

Why this interface exists

The consumer is any AO component that starts a long-running subprocess whose lifetime is bounded by the daemon. That consumer should not need to remember five separate concerns:

  1. register the PID for crash recovery,
  2. unregister on child exit,
  3. forward parent signals,
  4. kill process groups or Windows trees correctly, and
  5. escalate after a bounded grace period.

spawnManagedDaemonChild() is the formal interface because it aligns with that consumer: “I am starting a daemon-owned process; make its lifecycle match the daemon.” Lower-level APIs remain available only for integration points where AO cannot be the direct spawner.

Operational invariants

  • Invariant: a daemon-owned long-running child is either managed or explicitly outside the daemon lifecycle.
  • Invariant: ao stop does not rely solely on the daemon being alive; it can sweep the registry itself.
  • Invariant: Ctrl+C waits for the daemon shutdown sequence rather than racing a fallback exit.
  • Invariant: fallback signal handling still waits and escalates for simpler managed-child consumers.
  • Boundary: uncatchable parent death can still skip in-process cleanup; the registry and startup orphan sweep are the recovery path.
