Mobile Access for Agent Orchestrator

Replace the Tailscale-only mobile workflow with a single command — ao mobile — that spins up an HTTPS tunnel, gates the dashboard with an auto-generated token + cookie session, and prints a QR code. Scoped narrowly: single-user, opt-in, tunnel-only. Borrowed shape from OpenCode but does not copy their security posture — basic auth + plain HTTP is unsafe for a tool whose blast radius is "shell on the user's dev machine."

Revision note. This doc was rewritten after a Codex review caught three showstoppers in the original draft: (1) Next.js middleware does not gate WebSocket upgrades, (2) browsers do not reliably attach Authorization: Basic to new WebSocket() (especially on iOS Safari), and (3) user-chosen passwords combined with an in-memory rate limiter is not "brute-force resistant" for an internet-exposed RCE surface. See Codex review applied for the full list of changes.

Why not what we have today

Tailscale works, but it has friction that keeps people from actually using AO from their phone: the user must install a client on every device, share-with-a-teammate requires inviting them to a tailnet, and Tailscale itself becomes a dependency you may not want for a quick check-in from the couch. The bar is "open a URL, type a password, see your sessions."

The OpenCode model gets that part right — one command, one password, browser works. Their docs leave the transport unspecified (so it ends up plain HTTP for many users) and have no rate limit / origin pinning, which is the part we won't copy.

Subcommands

CommandWhat it does

No --no-tunnel / LAN mode in V1. Codex review flagged it as the most dangerous surface (binding 0.0.0.0 on hostile wifi = anyone on the LAN gets RCE). If a real use case appears we revisit it behind an explicit --lan-i-accept-the-risk flag with a printed warning.

Threat model — what V1 defends against. Passive snooping on hostile wifi (TLS); casual port-scanners (no inbound port); online token brute-force (128-bit + lockout-then-revoke); WebSocket hijack from a malicious page (HttpOnly SameSite=Strict cookie + WS upgrade re-check); server-log leakage of the credential (URL fragment delivery); curl-driven attacks that fake Origin (we don't depend on origin as the primary gate).
What V1 does not defend against. Phishing of the tunnel URL + token — the strongest realistic attack and the main reason V2 should ship Cloudflare Access SSO. A fake page styled as the AO mobile entry can capture the fragment token; users have no UI cue to distinguish real from fake. A malicious browser extension on the phone reading the URL fragment before the JS exchange (HttpOnly blocks reading the cookie afterward, but not the fragment) or operating the page DOM while the user is signed in. An attacker who has already compromised the user's host machine. A stolen unlocked phone with the cookie still valid (mitigation: ao mobile revoke). cloudflared binary tampering / PATH hijack (mitigation: pinned-checksum verify). Cloudflare itself, which terminates TLS and observes connection metadata. The fail-closed lockout being used as a remote DoS lever (see Security checklist row). These are documented in the README, not papered over.

Auth flow (corrected)

  1. ao mobile generates a 128-bit token. Stores sha256(token) + a persistent HMAC signing key at ~/.agent-orchestrator/mobile.json (mode 0600). Persisting the key means phone sessions survive ao stop / ao start --mobile cycles; only ao mobile revoke kills them.
  2. If a dashboard is already running without mobile mode: refuse with instructions. Otherwise start it bound to 127.0.0.1 with --mobile-origin as a CLI arg. No post-hoc env mutation.
  3. cloudflared spawns; stderr is parsed for the URL.
  4. Phone opens https://<tunnel>/m#t=<token>. The fragment is never sent to the server.
  5. /m page JS POSTs the token to /api/mobile/login. That route runs on the Node runtime (it touches the file-locked counter and session list). Server validates via constant-time hash compare, issues an HMAC-signed HttpOnly Secure SameSite=Strict cookie with 24h TTL. history.replaceState drops the fragment.
  6. Subsequent HTTP requests carry the cookie. Edge middleware verifies signature + expiry on every request — no fs reads, so revocation is enforced by the Node-runtime mutating routes and the WS gate, not by middleware.
  7. WebSocket upgrade handler (Node runtime) reads the cookie, runs the full verifier including store.isRevoked, requires a matching Origin, and rejects with HTTP 401 before handleUpgrade. After accept, a 60s re-verify interval runs per socket: on revoke or expiry, close 4001 revoked / 4002 session_expired.

Cookie verifiers — split by runtime

Two functions, two runtimes. The Edge-safe variant is stateless (signature + expiry only) so the Edge bundle never imports anything that touches fs. The Node variant calls the Edge variant first, then adds the revocation check. 

// packages/web/src/lib/mobile-auth-edge.ts — Edge-safe, no fs imports
import { timingSafeEqualHex, b64urlDecode } from "./crypto-edge";

export type Session = { id: string; iat: number; exp: number };
const MAX_COOKIE_BYTES = 4096;

// Stateless: signature + expiry only. Used by middleware.
export async function verifyCookieEdge(
  cookieHeader: string | null,
  hmacKey: CryptoKey,
): Promise<Session | null> {
  try {
    const raw = parseCookie(cookieHeader, "ao_mobile");
    if (!raw || raw.length > MAX_COOKIE_BYTES) return null;

    const parts = raw.split(".");
    if (parts.length !== 2) return null;
    const [body, sig] = parts;
    if (!body || !sig) return null;

    const expected = await hmacHex(hmacKey, body);
    if (!(await timingSafeEqualHex(sig, expected))) return null;

    const decoded = b64urlDecode(body);
    const session = JSON.parse(decoded);
    if (typeof session?.id !== "string"
     || typeof session?.iat !== "number"
     || typeof session?.exp !== "number") return null;

    if (session.exp <= Date.now()) return null;
    return session as Session;
  } catch {
    return null;
  }
}
// packages/web/src/lib/mobile-auth-node.ts — Node-only, may import the store
import { verifyCookieEdge, Session } from "./mobile-auth-edge";
import type { MobileStore } from "@aoagents/ao-core";

// Full check: stateless verify + revocation lookup. Used by login route's
// post-auth flows, mutating /api/* routes, and the WS upgrade gate.
export async function verifyCookieFull(
  cookieHeader: string | null,
  hmacKey: CryptoKey,
  store: MobileStore,
): Promise<Session | null> {
  const session = await verifyCookieEdge(cookieHeader, hmacKey);
  if (!session) return null;
  if (await store.isRevoked(session.id)) return null;
  return session;
}

HMAC key delivery: the dashboard process reads the key from mobile.json at startup and injects it into process.env.AO_MOBILE_COOKIE_KEY before Next boots. Next.js propagates process.env into the Edge runtime at startup, so middleware can read it without filesystem access. The Node-runtime side reads the same value from env or from the store directly.

WebSocket upgrade gate + process-wide re-verify

// In the existing WS server bootstrap. Imports liveSockets from mobile-runtime.ts
// so all upgrade paths share the same Map within this process.
import { liveSockets, store, hmacKey, expectedOrigin } from "./mobile-runtime";
import { verifyCookieFull } from "./mobile-auth-node";

const WS_PATH_ALLOWLIST = new Set(["/ws/terminal", "/ws/events"]);

httpServer.on("upgrade", async (req, socket, head) => {
  try {
    if (!isMobileEnabled()) {
      // Loopback assertion already enforced once at bootstrap; not repeated here.
      return wss.handleUpgrade(req, socket, head, (ws) => wss.emit("connection", ws, req));
    }

    const path = new URL(req.url ?? "/", "http://x").pathname;
    if (!WS_PATH_ALLOWLIST.has(path)) return reject(socket, 404, "unknown_path");

    const session = await verifyCookieFull(req.headers.cookie ?? null, hmacKey, store);
    if (!session) return reject(socket, 401, "no_session");

    const origin = normalizeOrigin(req.headers.origin);
    if (!origin || origin !== expectedOrigin) {
      return reject(socket, 403, "origin_mismatch");
    }

    wss.handleUpgrade(req, socket, head, (ws) => {
      let sockets = liveSockets.get(session.id);
      if (!sockets) liveSockets.set(session.id, (sockets = new Set()));
      sockets.add(ws);
      // Tag the socket so the scanner below knows the bound session.
      (ws as any)._aoSession = session;
      ws.on("close", () => sockets!.delete(ws));
      wss.emit("connection", ws, req);
    });
  } catch {
    reject(socket, 500, "internal");
  }
});

function reject(socket: Socket, code: number, reason: string) {
  socket.write(`HTTP/1.1 ${code} ${STATUS[code]}\r\n\r\n`);
  socket.destroy();
}

// One process-wide scanner — not a per-socket setInterval. Reads from the file
// store every tick so `ao mobile revoke` (a separate CLI process that just
// writes mobile.json) takes effect within ~60s without IPC.
//
// This is what makes the doc's "revoke kicks live sockets within 60s" claim
// honest: revoke is file-mediated, not in-process.
const REVERIFY_INTERVAL_MS = 60_000;
const reverifyTimer = setInterval(async () => {
  const now = Date.now();
  for (const [sessionId, sockets] of liveSockets) {
    const revoked = await store.isRevoked(sessionId);
    for (const ws of sockets) {
      const session = (ws as any)._aoSession as Session | undefined;
      if (revoked) ws.close(4001, "revoked");
      else if (session && session.exp <= now) ws.close(4002, "session_expired");
    }
  }
}, REVERIFY_INTERVAL_MS);
reverifyTimer.unref();   // don't keep process alive on shutdown

Revoke semantics — honest framing: ao mobile revoke is a separate CLI process. It cannot directly close sockets in the dashboard process. It writes a revocation record to mobile.json; the dashboard's process-wide scanner above reads from disk every 60s and closes any matching live sockets. Worst-case window: ~60s. Tightening below that would require an admin HTTP endpoint or Unix socket from CLI to dashboard — out of scope for V1.

Login route + lockout

// packages/web/src/app/api/mobile/login/route.ts
export const runtime = "nodejs";   // fs access required
const MAX_BODY_BYTES = 1024;

export async function POST(req: Request) {
  // Content-type with charset suffix is valid (e.g. "application/json; charset=utf-8").
  const ct = (req.headers.get("content-type") ?? "").toLowerCase();
  if (!ct.startsWith("application/json")) {
    return new Response("bad content-type", { status: 415 });
  }
  // Sec-Fetch-Site is absent on older browsers; only reject when present and wrong.
  const sfs = req.headers.get("sec-fetch-site");
  if (sfs && sfs !== "same-origin") {
    return new Response("bad origin", { status: 403 });
  }

  // Cap while reading the stream — content-length can be missing on chunked
  // bodies, so a header-only check is bypassable.
  const raw = await readWithCap(req.body, MAX_BODY_BYTES);
  if (raw === "OVERSIZE") return new Response("body too large", { status: 413 });

  const store = await getMobileStore();
  if (await store.isLocked()) {
    return new Response("locked — run `ao mobile revoke` on host", { status: 423 });
  }

  let token = "";
  try {
    const body = JSON.parse(raw);
    if (typeof body?.token === "string" && body.token.length <= 256) {
      token = body.token;
    }
  } catch { /* token stays "" */ }

  const ok = token !== "" && (await store.verifyToken(token));
  if (!ok) {
    await store.recordFailure();              // file-locked atomic; ≥10 → store.lock()
    return new Response("invalid", { status: 401 });
  }

  const session = await store.issueSession({ ttlMs: 24 * 3600 * 1000 });
  const cookie = await signSession(session, hmacKey);   // base64url(json).hmacHex
  return new Response(null, {
    status: 204,
    headers: {
      "Set-Cookie": `ao_mobile=${cookie}; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=86400`,
    },
  });
}

// readWithCap: pulls chunks from the body and aborts if the running total
// exceeds `limit`. Returns the string on success, or the literal "OVERSIZE".
async function readWithCap(body: ReadableStream | null, limit: number) {
  if (!body) return "";
  const reader = body.getReader();
  const chunks: Uint8Array[] = [];
  let total = 0;
  while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    total += value.byteLength;
    if (total > limit) { reader.cancel(); return "OVERSIZE"; }
    chunks.push(value);
  }
  return new TextDecoder().decode(Buffer.concat(chunks.map((c) => Buffer.from(c))));
}

cloudflared lifecycle

  1. Locate cloudflared on PATH. Binary checksum verification deferred to V1.5 — V1 prints a one-line note that PATH is trusted.
  2. Spawn cloudflared tunnel --url http://127.0.0.1:$PORT --no-autoupdate.
  3. Stream stderr; match https://[a-z0-9-]+\.trycloudflare\.com for the URL. Treat any URL on a different host as a hard error.
  4. Pass the normalized URL to the dashboard as --mobile-origin at start, never as post-hoc env on a running process. If a dashboard is already running without mobile mode, ao mobile refuses with the instruction to ao stop && ao start --mobile first. No silent restart.
  5. Render QR + URL with the token in the fragment. Block on the cloudflared process.
  6. On Ctrl+C: invalidate all sessions in the mobile-store, send SIGTERM to cloudflared, wait 5s, SIGKILL.
  7. If ao mobile runs while ao start isn't already up, start the dashboard inline first (same code path as ao start) so the loopback bind is enforced from boot.

What we are explicitly not building in V1

Deferred to V1.5 (eng-review scope reduction)

  • Audit log to mobile-audit.jsonl. V1 relies on stdout + cloudflared access logs. Forensic capability lands in V1.5.
  • cloudflared binary checksum verification. V1 trusts PATH; PATH-hijack is documented as not-defended-in-V1. Pinned-checksum verify ships in V1.5.
  • ao mobile install subcommand. V1 prints brew/apt instructions on missing binary and exits.

Deferred indefinitely

  • LAN mode (--no-tunnel). Removed entirely. Binding 0.0.0.0 on a typical laptop wifi exposes the RCE surface to every device on the network. Returns later only with a loud opt-in flag.
  • Per-device passkeys / WebAuthn. V1 has session-level revocation, not device-level.
  • Cloudflare Access SSO. Documented as a hardening path; no first-class support yet.
  • Mobile-optimized UI. Dashboard responds to small viewports adequately. Separate track.
  • Push notifications. Existing notifier plugin slot covers this.
  • Multiple concurrent tunnels. One per machine.
  • ngrok fallback. Cloudflare-only.

Eng review applied

Plan went through /plan-eng-review after the second Codex pass. Five decisions taken; folded into the doc above.

Eng-review issueDecision
Issue 1. Login route writes to filesystem; Edge runtime can't do that. Pin login route + any mobile route that writes to Node runtime (export const runtime = "nodejs"). Middleware stays Edge for cookie-signature verify only (no fs).
Issue 2. What if dashboard is already running without mobile mode? ao mobile refuses with ao stop && ao start --mobile instruction. No silent restart, no orphaned desktop browser tabs.
Issue 3. HMAC key lifetime unspecified. Persist HMAC signing key in mobile.json alongside token hash (mode 0600). Phone stays signed in across ao stop / ao start --mobile; only revoke kills sessions.

Scope reduction. Plan touched 13 files / 3 services — past the complexity gate. Defer audit log, cloudflared checksum verify, and ao mobile install to V1.5. All Codex hard-blockers stay in V1.

Codex review applied

The first draft of this doc went to a Codex consult. It found three showstoppers and several smaller bugs. Each item below names what changed and where to look in the doc above.

Codex findingWhat changed
"Next.js middleware does not protect raw WebSocket upgrade paths." WS auth is now enforced inside the upgrade event handler before handleUpgrade — see WebSocket upgrade gate. Doc no longer claims middleware covers WS.
"Browsers do not let JS set arbitrary Authorization headers on new WebSocket(...). iOS Safari unreliable for Basic + WS." Dropped HTTP basic for the dashboard entirely. Auth is now: token-via-fragment → POST /api/mobile/login → HttpOnly SameSite=Strict cookie. The cookie rides with the WS upgrade automatically. Verified path is the same on every platform.
"In-memory rate limiter resets on restart, per-process only, multi-IP bypass, unbounded Map = memory DoS, cf-connecting-ip spoofable in LAN mode." Replaced with daemon-wide failure counter persisted to mobile.json. Hits 10 → feature locks until ao mobile revoke. Not per-IP (spoofable). LAN mode removed in V1, sidestepping the cf-connecting-ip spoofing concern.
"timingSafeEqual is not a JS built-in; Node's throws on unequal-length buffers; Edge runtime may not have Buffer/crypto." All comparison goes through a single timingSafeEqualHex helper backed by crypto.subtle (Edge-safe), with explicit length-mismatch handling that always returns false in constant time.
"atob(...).split(":") breaks passwords containing colons." Moot now — no basic-auth parsing.
"AO_MOBILE_ORIGIN can't be set on an already-running dashboard process." Origin (and the mobile-store path) is passed at dashboard start as a CLI arg, not env. ao mobile restarts the dashboard inline if needed.
"Origin pinning helps against browser cross-site WS abuse, not curl/native clients (RFC 6455)." Doc now states explicitly that Origin is defense-in-depth. The primary WS gate is the signed cookie + revocation check.
"User-chosen 16-char password is meaningless. Require generated 128-bit secret." Token is generated by ao mobile via crypto.randomBytes(16). There is no AO_MOBILE_PASSWORD env var any more.
"--no-tunnel binding 0.0.0.0 exposes RCE to the LAN." Removed in V1. Dashboard hard-binds to 127.0.0.1 when mobile is enabled.
"cloudflared stdout is not an audit log; PATH hijack possible." Both deferred to V1.5 in the eng-review scope reduction. V1 ships with stdout + cloudflared logs and trusts PATH; V1.5 adds the redacted JSONL audit log and pinned-checksum verification.
"No CSRF story for HTTP mutating endpoints." SameSite=Strict on the session cookie prevents cross-site form posts. All mutating routes also require a POST + cookie + JSON body content-type.
"Phishing of the tunnel URL." (Codex called this out as a hard limit.) Documented in the threat-model callout as not-defended in V1. SSO via Cloudflare Access is the V2 fix; doc notes how to wire it up for users who care.

Second pass (post-rewrite)

Codex re-reviewed the rewrite and surfaced a smaller set of correctness + claim-honesty issues. Each is folded into the doc above.

Codex finding (2nd pass)What changed
"Middleware as written would block /api/mobile/login and /m — first login deadlocks." Implementation plan → middleware.ts now spells out the public allowlist (/m, /api/mobile/login, static assets).
"Cookie parser can throw on attacker junk → 500s. Should fail closed." Cookie verifier now wraps everything in try/catch, returns null on any malformed input. Verification tests cover this explicitly.
"Cookie body should be base64url, not raw base64. Enforce exactly 2 dot-parts. Cap cookie size." Verifier uses b64urlDecode, asserts parts.length === 2, caps at 4KB. Sign path emits base64url too.
"Cookie expiry uses <; should be <=." Fixed in the verifier sketch.
"Threat model wrongly says HttpOnly cookie is readable by extension." Reworded: extension can read the URL fragment before JS exchange, or operate the page DOM. HttpOnly does block direct cookie reads.
"WS revoke does not actually close live sockets." WS gate now tracks Map<sessionId, Set<WebSocket>>. killSessionSockets() is called by ao mobile revoke and tunnel teardown. Verification table updated.
"origin && origin !== expected allows missing Origin to pass." WS gate now requires Origin in mobile mode and rejects missing as 403. expectedOrigin is normalized (scheme + lowercase host, no trailing slash).
"Async verifier throw can leave the socket hanging." Whole upgrade handler is in try/catch; errors close with 500 and audit-log ws_error.
"No path allowlist on WS upgrades." WS_PATH_ALLOWLIST added; unknown paths get 404.
"File-backed counter is a remote DoS lever — don't sell it as 'brute-force resistant.'" Security checklist row reframed: brute-force resistance comes from the 128-bit token; the lockout is "fail-closed by design, also a DoS lever," documented honestly. Threat model callout names this explicitly.
"Login route missing body cap, content-type check, atomic counter, file lock." Login route sketch now enforces 1KB cap, application/json only, Sec-Fetch-Site check; failure counter and session list use file-locking. Concurrency test added.
"'Or via mobile-store file' for origin is hand-wavy." cloudflared lifecycle is now unambiguous: origin is a CLI arg at dashboard start. If a dashboard is already running, restart or refuse. No watch-read fallback.
"CSRF claim 'POST + JSON content-type' is too strong." Security checklist now cites SameSite=Strict + Sec-Fetch-Site: same-origin as the actual defense, with content-type as enforcement (not defense in itself).
"No clickjacking headers, no CSP on /m." Added: frame-ancestors 'none' globally, tight per-route CSP on /m since it handles the fragment token.
"Audit log: prompt text could leak secrets — decide explicitly." Audit log itself deferred to V1.5 (eng-review scope reduction). The redaction policy stands when V1.5 lands: metadata only (kind, session id, length, timestamp), no prompt text, no tokens, mode 0600.
"Read-only mode is V2 → V1 mobile has full RCE-equivalent power." CLI banner now prints an explicit warning when ao mobile starts. README will repeat it.

Third pass (post eng-review)

Codex re-reviewed the eng-review delta and caught one structural problem and a list of drift / correctness fixes. All folded in.

Codex finding (3rd pass)What changed
Structural: the cookie verifier said it ran in middleware AND ws-auth, but middleware can't call store.isRevoked. Internally inconsistent and would pull MobileStore (and fs) into the Edge bundle. Split into two functions, two files: verifyCookieEdge in mobile-auth-edge.ts (signature + expiry only, no fs imports) and verifyCookieFull in mobile-auth-node.ts (calls Edge variant then adds revocation). Edge bundle is provably fs-free.
"How does Edge middleware get the persisted HMAC key without fs?" Dashboard process reads the key from mobile.json at boot and injects it into process.env.AO_MOBILE_COOKIE_KEY. Next.js propagates process.env into the Edge runtime at startup.
Structural: ao mobile revoke is a separate CLI process; importing mobile-runtime.ts from it gets a fresh empty liveSockets Map. killSessionSockets() from CLI was a no-op. Revoke is now file-mediated: CLI writes a revocation record to mobile.json; the dashboard's process-wide 60s scanner reads it and closes matching sockets. Doc framing changed from "kicks live sockets" to "kicks live sockets within ~60s." Honest. No IPC needed in V1.
"WS upgrade-gate pseudocode said periodic re-verify but didn't actually create the interval. Per-socket intervals would leak." Sketch now creates one process-wide setInterval that scans liveSockets every 60s. Timer is unref'd for clean shutdown.
"mobile-runtime.ts overclaims process-wide singleton via Node module cache." Doc reframed: mobile-runtime.ts owns process-local WS state inside the dashboard process only. mobile.json is the source of truth across processes (CLI ↔ dashboard, Node ↔ Edge).
"CLI prints https://host/#t=…; auth flow needs /m#t=…. Root fragment won't trigger the login page." CLI output sample fixed to https://host/m#t=…. Snapshot test added to lock it.
"content-type === 'application/json' rejects valid application/json; charset=utf-8." Login route uses startsWith('application/json') on the lowercased header.
"Missing content-length → 0 → chunked body bypasses 1KB cap." Body cap is now enforced while reading the stream via a readWithCap helper that aborts on overflow.
"/m CSP script-src 'self' may break Next's inline runtime." /m is now declared as a static client component with a per-request CSP nonce; <noscript> fallback for JS-disabled phones.
"Verification row 'middleware blocks all /api/*' missed the public allowlist." Test description updated to except /api/mobile/login and /m; allowlisted-paths-pass-through is now an explicit assertion.
"cloudflared lifecycle says 'stops it and starts a fresh one — or refuses with --no-restart' — contradicts the eng-review 'always refuse' decision." Lifecycle now says always-refuse, no --no-restart flag.
"Files-table still listed install; cloudflared.ts row still claimed checksum verify on first use." Both removed; both are V1.5 per scope reduction.
"Codex-review-applied table still claimed audit log + checksum were V1." Reworded to reflect V1.5 deferral.
Audit references in code sketches contradicted the V1.5 deferral. Stripped audit.log calls from the WS upgrade gate and login route sketches; mobile-runtime.ts row no longer claims to own an audit handle.
Honest revoke window. V1 revoke is file-mediated and takes effect in up to 60 seconds on already-open WebSocket connections. The doc previously implied immediate. New requests (HTTP and WS) are blocked the moment mobile.json is rewritten — the lag only applies to sockets that were authorized before the revoke. Tightening this would require an admin HTTP endpoint or Unix socket from CLI to dashboard; out of scope for V1.
